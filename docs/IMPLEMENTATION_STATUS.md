# FARP 实现状态

本文记录 FARP 仓库已经落地的能力、对应安全边界和后续工作，防止协议愿景与当前代码状态混淆。

更新日期：2026-07-17

当前阶段：v0.1 安全核心

## 本阶段范围

本阶段只建立后续网络协议必须依赖的本地安全基础：稳定身份、设备证书、确定性签名、永久贡献状态模型、统一限流、单一 SQLite schema 和安全回归测试。

本阶段没有实现真实聊天网络、DHT、离线 Relay、随机见证、双棘轮或 PUK 恢复。

## 已实现能力

### 1. 账户与设备身份分离

`farp/identity` 已将身份拆分为三类密钥：

- 账户根 Ed25519 密钥：定义稳定的 FARP 账户 ID，并为设备证书签名。
- 当前设备 Ed25519 密钥：签署消息信封和服务授权，不直接暴露账户根私钥。
- X25519 协商密钥：为后续端到端会话密钥协商预留。

设备证书绑定以下字段：

```text
account_pubkey
device_pubkey
agreement_pubkey
previous_device_pubkey
device_id
device_epoch
issued_at
```

证书由账户根密钥签名。加载身份时会重新推导公钥并验证私钥、设备证书、设备 ID 和账户 ID 是否一致，篡改任一部分都会导致加载失败。

### 2. 本地身份加密持久化

`internal/identitystore` 已实现：

- 守护进程首次启动时生成身份，后续重启复用同一账户。
- 私钥包使用 AES-256-GCM 加密后存入 SQLite。
- 账户公钥被作为附加认证数据，防止数据库中的公钥与密文身份包被调换。
- 设备包装密钥文件和 SQLite 文件设置为仅当前用户可读写。
- 身份密文、认证标签或数据库绑定信息被修改后，加载会失败，而不是静默生成新账户。

当前包装密钥仍保存在数据目录中的 `device.key`，这是接入操作系统密钥库和 PUK 机制之前的过渡实现。它能防止数据库文件单独泄露，但不能抵御整个数据目录同时被复制。

### 3. 确定性协议签名

`farp/protocol` 提供带域分离和长度前缀的确定性编码器，避免字符串拼接、字段歧义和不同对象之间的签名重用。

`farp/messaging` 的消息信封签名目前覆盖：

```text
version
message_id
task_id
sender_account_pubkey
recipient_pubkey
device_certificate
ephemeral_pubkey
ciphertext
nonce
created_at
expires_at
max_hops
message_type
```

验证过程会检查设备证书、账户绑定、签名、密文大小、随机数长度、最大跳数、有效期、最大生命周期和未来时钟偏差。修改密文、接收者、期限或其他安全字段都会使签名失效。

服务授权 `ServiceAuthorization` 使用相同原则，并强制要求：

- 消费金额必须大于 0。
- 任务 ID、服务节点、随机数、签发时间和过期时间全部进入签名。
- 授权最长有效 24 小时，不能通过负数金额增加配额。

### 4. 永久贡献状态模型

旧的 `farp/ledger` 和 7 天贡献率冻结逻辑已经删除，替换为 `farp/contribution`。

当前本地状态明确拆分为：

```text
SpendableCredit
LifetimeContribution
BurnedCredit
StateVersion
```

已经落实的规则：

- 只有显式调用有效贡献奖励才增加配额，单纯在线不会增加。
- 配额不随时间衰减，不存在近期贡献率冻结。
- 赚取、消费和销毁金额必须为正数。
- 余额不足时拒绝消费或销毁。
- 检查整数溢出。
- 每次有效变更递增账户状态版本。
- 状态读写具备进程内并发保护。

当前贡献状态仍是内存投影，尚未与随机见证证明和 SQLite `credit_states` 表连接，因此守护进程重启后的配额恢复尚未实现。

### 5. 统一硬限流

`farp/ratelimit` 对同一账户的不同公共操作使用同一个令牌桶：寻址、暂存、转发、见证等操作不能通过更换操作类型绕过额度。

节点可在每分钟 5 至 10 次之间配置容量，代码会强制夹紧上下界：

- 低于 5 自动按 5 处理。
- 高于 10 自动按 10 处理。
- 已创建的账户令牌桶也会同步新容量。
- 荣誉、历史贡献和配额余额不能突破该硬件保护。

该应用层限流不能替代防火墙、连接数限制和网络层 DDoS 防护。

### 6. 单一 SQLite schema

数据库 schema 已移动到 `internal/db/schema.sql` 并通过 `go:embed` 编译进程序，删除了原有的重复 schema 来源。

SQLite 驱动已从依赖 CGO 的 `go-sqlite3` 切换到纯 Go 的 `modernc.org/sqlite`，便于 Windows、低配节点和交叉编译环境使用。

schema version 2 包含：

- `local_identity`：加密身份包。
- `local_contacts`：本地联系人和加密棘轮状态。
- `local_messages`：通信双方本地保存的消息密文。
- `outbox_items`：本地待发送队列。
- `route_cache`：有 TTL 的加密路由缓存。
- `relay_cache`：中继节点保存的不透明密文、任务 ID 和下一跳令牌。
- `contribution_receipts`：本地贡献证明。
- `credit_states`：账户贡献状态投影。
- `witness_markers`：滚动见证最小状态。
- `local_device_state`：单活设备状态预留。
- `local_rate_quota`：持久化限流状态预留。

`relay_cache` 不保存发送者或接收者公钥。发送者本机的 `outbox_items` 保存接收者公钥属于通信端本地数据，不属于中间节点数据。

旧版表不会被破坏性删除，schema version 2 使用新表名，便于后续显式迁移。

### 7. 守护进程与本地 API

`cmd/daemon` 现在会：

- 创建并初始化 SQLite。
- 加载或创建稳定的加密身份。
- 初始化贡献状态和统一限流。
- 只在 `127.0.0.1` 的随机端口启动 API。
- 在收到退出信号时优雅关闭 HTTP 服务和数据库。
- 对尚未实现的 `--relay` 和 `--dht-server` 模式明确返回错误，不再伪装成可用功能。

`internal/api` 已删除通配符 CORS，网页不能直接跨域调用本地节点；同时加入读取、写入、空闲和请求头超时。

本地 API 尚未加入进程级认证令牌，因此同一台设备上的其他本地进程仍可能访问它。实现 CLI/GUI 通信时需要增加随机本地令牌或操作系统 IPC 权限验证。

## 已删除或替换的旧实现

- 删除会被负数消费绕过、带 7 天冻结逻辑的 `farp/ledger`。
- 删除未覆盖密文且验证函数固定成功的旧消息验证逻辑。
- 删除重复的 `db/schema.sql` 内嵌副本。
- 删除尚未接入生成流程、会形成第二套协议定义的旧 `envelope.proto`。
- 删除本地 API 的 `Access-Control-Allow-Origin: *`。
- 守护进程不再在每次启动时生成新身份。

## 测试与验证

当前自动化测试覆盖：

- 身份私钥包序列化和恢复。
- 设备证书字段篡改检测。
- 守护进程身份跨数据库重启保持一致。
- 加密身份密文篡改检测。
- 消息信封密文篡改、过期和签名验证。
- 服务授权金额和签名字段篡改检测。
- 永久贡献状态的赚取、消费、销毁、负数与溢出保护。
- SQLite schema 幂等初始化、版本号和禁止明文字段。
- 统一限流、跨操作累计和 5 至 10 次硬边界。
- 本地 API 状态字段和禁止开放 CORS。

验证命令：

```bash
go test -count=1 ./...
go vet ./...
```

以上命令在 2026-07-17 的当前工作树中均已通过。

## 尚未实现

以下内容仍属于后续协议和工程阶段：

1. 威胁模型、消息状态机、账户状态机和滚动见证状态机定稿。
2. 双棘轮或等价的真实端到端消息加密。
3. 本地批量队列、P2P 直连发送、ACK 和去重。
4. 离线多副本 Relay、逐跳令牌、TTL 清理和交付证明。
5. 两层随机见证、跨域分散、三分之一阈值和极端离线降级。
6. 贡献证明验证、配额持久化、消费交接和旧见证退休。
7. 单活设备迁移、上一设备恢复、操作系统密钥库和 PUK。
8. DHT、NAT 穿透、弱网路由、故障注入和大规模恶意节点模拟。

在这些能力完成并经过独立安全审计前，FARP 不能用于生产通信或重要数据。
